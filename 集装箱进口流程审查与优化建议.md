# 集装箱进口全流程审查与系统逻辑优化建议

## 一、审查结论

当前系统已经具备“Manifest 导入 -> 船舶/集装箱/作业任务入库 -> 后台自动作业流 -> 岸桥卸船 -> AGV 转运 -> 场桥入堆”的基础闭环。也就是说，系统可以完成进口箱从船上到堆场的前半段生命周期，并且已有 `ship`、`container`、`task`、`yard`、`equipment` 等核心表作为数据支撑。

但是，需求中描述的完整进口流程还包括“海关/商检放行、提箱预约、闸口进场、堆场提箱、闸口出场、箱子离港”。这些后半段环节在当前系统中还没有形成独立数据模型和强校验逻辑，主要依赖人工修改集装箱状态，容易出现状态断层、越权提箱、错误放行、重复作业、异常无记录等风险。

因此，本报告建议将进口箱生命周期拆成三个层次管理：

1. **实物状态**：箱子在船上、已卸船、转运中、在场、提箱中、已离港。
2. **监管状态**：未申报、查验中、扣留、已放行。
3. **业务单证状态**：Manifest 已导入、卸船任务完成、预约已确认、闸口进场、提箱任务完成、出闸完成。

三类状态不能混在一个 `container.status` 字段中，否则会导致“箱子在场但未放行”“箱子已放行但未预约”“车辆已进闸但未提箱”等真实业务状态无法表达。

## 二、建议后的进口主流程

### 1. Manifest 导入

系统导入船舶任务清单，创建或更新以下数据：

- `ship`：船名、航次、ETA、ETD、泊位、船舶状态。
- `container`：箱号、箱型、尺寸、空重、危险品、冷藏、所属船舶、初始实物状态。
- `task`：自动生成卸船任务、水平运输任务、入堆任务，或先生成卸船任务，再由流程引擎按条件派生后续任务。
- 建议新增 `manifest` 和 `manifest_item`：记录导入批次、原始清单、导入人、导入时间、校验结果、重复箱/缺字段/格式错误等。

Manifest 导入完成后，集装箱初始状态建议为：

- 实物状态：`在船上`
- 监管状态：`未放行`
- 业务状态：`待卸船`

### 2. 船舶靠泊与卸船

船舶分配泊位并变更为 `已靠泊` 后，岸桥任务才允许开始。岸桥完成单箱卸船后：

- 箱子实物状态从 `在船上` 变为 `已卸船`。
- 卸船任务状态从 `进行中` 变为 `已完成`。
- 理货确认记录写入系统。
- 如箱号不符、残损、溢短卸，应转入异常流程，不应直接进入正常转运。

### 3. 水平运输

AGV 或内集卡接收运输任务后，系统应校验：

- 箱子是否已卸船。
- 箱号与任务箱号是否一致。
- 目标堆场是否可用。
- 岸桥前沿缓冲区是否有可提箱。
- 交接位置是否被占用。

运输开始后，箱子实物状态变为 `转运中`；到达堆场交接点后，运输任务完成。

### 4. 堆场入堆

场桥接收任务后，将箱子放入目标箱位。系统应更新：

- `container.yard`
- `container.area`
- `container.column`
- `container.layer`
- 实物状态：`堆场存储` 或 `在场`

建议新增 `yard_slot` 表，精确管理每一个 Bay/Row/Tier 是否占用、锁定、故障、禁用，而不是只靠 `container` 表的位置字段推断堆场占用。

### 6. 提箱预约

建议新增 `pickup_appointment` 表，字段包括：

- 预约号
- 箱号
- 车牌号
- 司机姓名和证件
- 预约时间窗
- 预约状态：`待确认`、`已确认`、`已进闸`、`已提箱`、`已出闸`、`已取消`、`已过期`
- 货代/客户信息

预约创建时必须校验：

- 箱子是否在场。
- 海关/商检是否放行。
- 是否已经存在未完成预约。
- 堆场是否具备提箱作业条件。
- 危险品、冷藏箱、超限箱是否需要特殊审批。

### 7. 闸口进场

外卡到达闸口后，系统应进行车牌、预约、箱号、司机身份、时间窗校验。校验通过后：

- 预约状态变为 `已进闸`。
- 生成闸口进场记录。
- 打印或下发岗位小票。
- 系统生成或激活堆场提箱任务。

如果任一条件不满足，应触发闸口拦截，不能靠人工记忆判断。

### 8. 堆场提箱

场桥根据提箱任务，将箱子吊装至外卡。完成后：

- 箱子实物状态建议变为 `提箱中` 或 `已装车待出闸`。
- 提箱任务状态变为 `已完成`。
- 原堆场箱位释放。
- 预约状态变为 `已提箱`。

该环节需要防止“一个箱子被多个预约或多个任务同时锁定”。建议使用箱级锁定字段或 `yard_slot.lock_status` 实现并发控制。

### 9. 闸口出场

外卡出场时，系统应再次核验：

- 当前车辆是否有已进闸预约。
- 当前车牌是否与预约车牌一致。
- 当前车载箱号是否与预约箱号一致。
- 海关状态是否仍为已放行。
- 提箱任务是否已完成。

校验通过后：

- 箱子实物状态变为 `已离港`。
- 预约状态变为 `已出闸`。
- 闸口出场记录完成。
- 进口箱流程结束。

## 三、状态流转闭环性风险与建议

| 风险点 | 当前表现 | 可能后果 | 系统逻辑优化建议 |
|---|---|---|---|
| `container.status` 同时承载实物、业务、监管含义 | 当前状态流包含 `在船上`、`已卸船`、`堆场存储`、`等待提箱`、`离港`，但没有海关、预约、闸口状态 | 箱子在场但未放行时，系统无法表达；可能被误提 | 将状态拆成 `physical_status`、`customs_status`、`appointment_status` 或至少新增独立字段 |
| 缺少 Manifest 批次状态 | 导入后只写入船舶、箱子、任务 | 无法追溯某个箱子来自哪次导入；重复导入难以审计 | 新增 `manifest`、`manifest_item`，记录导入批次、原始数据、校验结果 |
| 卸船任务完成不一定驱动运输任务 | 当前自动工作流可推进，但普通任务状态与箱子状态没有强绑定 | 操作员手动完成任务后，箱子状态可能仍停留在旧状态 | 在任务完成接口中按任务类型触发箱子状态迁移，并写入任务日志 |
| 船舶状态和箱子状态可能冲突 | 自动工作流最后会将船舶置为 `已离港`，但仍可能有未入堆或异常箱 | 船舶已离港但箱子还在异常或前沿缓冲区 | 船舶离港前校验该船所有箱子均为 `在场`、`异常闭环` 或 `短卸确认` |
| 缺少状态回退和异常挂起机制 | 当前状态多为单向推进 | 箱号不符、残损、海关扣留时只能人工改状态 | 增加 `异常挂起`、`待人工确认`、`已解除异常` 等状态，并建立异常处理表 |
| 箱位释放没有严格触发点 | `yard.used_capacity()` 通过未离港箱数量推断 | 已装车但未出闸的箱子仍可能占用堆场；或提前释放导致箱位冲突 | 在堆场提箱完成时释放 `yard_slot`，在闸口出场时更新箱子离港 |

## 四、数据与信息流承接风险与建议

| 环节 | 必需数据 | 当前系统覆盖情况 | 缺口与优化建议 |
|---|---|---|---|
| Manifest 导入 | 箱号、箱型、空重、危险品、冷藏、目标堆场、所属船舶 | 已能导入箱号、箱型、状态、堆场、区域、列层、危险品、冷藏 | 建议补充尺寸、重量、卸船序号、提单号、封志号、客户/货代、监管属性 |
| 岸桥卸船 | 卸船顺序、桥吊编号、理货确认 | 当前可生成任务并分配设备 | 建议新增理货确认字段：实卸箱号、残损标记、理货员、确认时间 |
| 水平运输 | 起点、终点、车辆/AGV、交接点、缓冲区容量 | 自动工作流已有 AGV 转运和缓冲区容量概念 | 建议将缓冲区和交接点持久化为表，而不是只存在内存逻辑中 |
| 堆场入堆 | 目标箱位、场桥、箱位占用、箱类限制 | 已能自动分配堆场位置 | 建议新增 `yard_slot`，并校验危险品、冷藏箱是否进入适配堆场 |
| 海关放行 | 放行状态、查验状态、放行指令号 | 当前未覆盖 | 新增 `customs_release` 或在 `container` 中增加监管字段 |
| 提箱预约 | 预约号、车牌、司机、时间窗、箱号 | 当前未覆盖 | 新增预约表和预约接口 |
| 闸口进出 | 车牌识别、箱号识别、闸口记录、拦截原因 | 当前未覆盖 | 新增 `gate_transaction` 表，进闸/出闸分别记录 |
| 堆场提箱 | 提箱任务、外卡车、原箱位释放 | 当前只支持通用任务 | 新增 `提箱作业` 类型，并绑定预约和闸口进场记录 |

## 五、异常与边界场景风险与建议

### 1. 海关未放行仍尝试进闸

**风险**：如果系统只校验箱子是否在场，而不校验海关状态，司机可能提前进闸提箱。

**建议**：

- 闸口进场接口必须校验 `customs_status == 已放行`。
- 未放行时返回拦截原因：`海关未放行，禁止进场提箱`。
- 拦截记录写入 `gate_exception` 或 `alarm`，便于查询。

### 2. 卸船时发现箱号不符

**风险**：Manifest 中的箱号与实际吊下箱号不一致，若系统仍按计划箱号推进，会造成串箱。

**建议**：

- 岸桥/理货确认时输入或扫描实际箱号。
- 若实际箱号不存在于 Manifest，创建 `溢卸异常`。
- 若 Manifest 有计划箱但未卸到，标记 `短卸异常`。
- 异常未解除前，不允许自动入堆或提箱。

### 3. 卸船时发现残损

**风险**：残损箱进入正常流程后，后续责任无法界定。

**建议**：

- 新增 `damage_status`：`正常`、`轻微残损`、`严重残损`、`待验残`。
- 新增 `damage_report`，记录照片、残损位置、发现环节、责任方、处理意见。
- 严重残损箱自动进入指定异常堆场或查验区。

### 4. 危险品箱或冷藏箱堆场分配错误

**风险**：危险品箱进入普通堆场，冷藏箱未接电，会产生安全和货损风险。

**建议**：

- Manifest 导入时识别危险品、冷藏属性。
- 箱位分配时按堆场用途过滤。
- 冷藏箱入堆后生成接电确认任务。
- 危险品箱增加隔离规则和堆存限制。

### 5. 外卡车提错箱

**风险**：闸口或堆场只看车牌，不核验箱号，可能把错误箱装车。

**建议**：

- 提箱任务必须绑定预约号、车牌号、箱号。
- 堆场提箱前扫描箱号，与任务箱号不一致时禁止吊装。
- 闸口出场再次 OCR/人工确认箱号和车牌。

### 6. 预约时间外进闸

**风险**：大量车辆集中到达，造成闸口拥堵和堆场作业计划失衡。

**建议**：

- 预约表设置允许提前/延后时间，例如前后 30 分钟。
- 超窗到达时状态为 `预约超窗`，需要调度确认或重新预约。
- 闸口系统根据时间窗自动放行或拦截。

### 7. 重复预约同一箱

**风险**：同一箱被多个车牌预约，导致提箱任务冲突。

**建议**：

- 对 `pickup_appointment` 增加约束：同一箱号同一时间只能有一个未完成预约。
- 预约确认后将箱子标记为 `已预约锁定`。
- 取消或过期预约时释放锁定。

### 8. 堆场无可用箱位

**风险**：卸船已经开始，但系统找不到箱位，导致前沿缓冲区堵塞。

**建议**：

- Manifest 导入后先做箱位预分配和容量预校验。
- 若目标堆场容量不足，提前预警，不允许直接启动整船自动作业。
- 设置临时堆场或异常等待区。

### 9. 设备故障导致任务卡死

**风险**：岸桥、AGV、场桥故障后，任务一直保持 `进行中`。

**建议**：

- 任务分配前校验设备状态。
- 设备故障时自动释放未完成任务，转为 `待重新派工`。
- 新增任务超时监控，超过预计时间触发告警。

## 六、性能与并发隐患

| 并发场景 | 风险 | 优化建议 |
|---|---|---|
| 多艘船同时导入 Manifest | 箱号重复、船舶/任务批次混乱 | 使用事务处理导入；按 `manifest_id` 分批；箱号唯一冲突返回明确结果 |
| 多艘船同时启动自动工作流 | 泊位、设备、堆场箱位竞争 | 对泊位、设备、箱位增加锁定机制；工作流启动前做资源预占 |
| 大量箱子同时入堆 | `yard.used_capacity()` 每次实时统计可能变慢 | 维护 `yard_slot` 和堆场容量缓存；关键操作中增量更新 |
| 多辆外卡同时进闸 | 同一箱可能被重复放行 | 预约和箱子加唯一约束；进闸时原子更新预约状态 |
| 后台线程状态只存在内存 | Flask 重启后工作流状态丢失 | 将工作流实例写入 `workflow_run` 表，阶段进度持久化 |
| SQLite 承载高并发写入能力有限 | 多用户写入、导入、工作流同时执行时容易锁库 | 课程项目可保留 SQLite；如扩展真实系统，建议迁移 MySQL/PostgreSQL |
| 前端轮询工作流状态 | 船舶多时请求量增加 | 可降低轮询频率，或按选中船舶查询；真实系统可用 WebSocket/SSE |

## 七、建议新增的数据表

### 1. `manifest`

用于记录每次导入批次。

| 字段 | 含义 |
|---|---|
| `id` | 主键 |
| `ship_id` | 船舶 ID |
| `file_name` | 文件名 |
| `imported_by` | 导入人 |
| `imported_at` | 导入时间 |
| `status` | `校验中`、`已导入`、`部分失败`、`已作废` |
| `error_count` | 错误条数 |

### 2. `manifest_item`

用于保存 Manifest 明细和导入校验结果。

| 字段 | 含义 |
|---|---|
| `manifest_id` | 导入批次 |
| `container_no` | 箱号 |
| `container_type` | 箱型 |
| `size` | 尺寸 |
| `discharge_seq` | 卸船顺序 |
| `target_yard` | 目标堆场 |
| `target_slot` | 目标箱位 |
| `check_result` | 校验结果 |

### 3. `yard_slot`

用于精确管理堆场箱位。

| 字段 | 含义 |
|---|---|
| `yard_id` | 堆场 |
| `area` | 区域 |
| `bay` | 贝位 |
| `row` | 排 |
| `tier` | 层 |
| `container_id` | 当前占用箱 |
| `slot_status` | `空闲`、`占用`、`锁定`、`禁用` |

### 4. `customs_release`

用于记录海关/商检放行。

| 字段 | 含义 |
|---|---|
| `container_id` | 箱子 |
| `customs_status` | 海关状态 |
| `inspection_status` | 商检状态 |
| `release_no` | 放行指令号 |
| `released_at` | 放行时间 |
| `hold_reason` | 扣留原因 |

### 5. `pickup_appointment`

用于管理提箱预约。

| 字段 | 含义 |
|---|---|
| `appointment_no` | 预约号 |
| `container_id` | 箱子 |
| `truck_plate` | 车牌 |
| `driver_name` | 司机 |
| `time_window_start` | 预约开始 |
| `time_window_end` | 预约结束 |
| `status` | 预约状态 |

### 6. `gate_transaction`

用于记录闸口进出。

| 字段 | 含义 |
|---|---|
| `appointment_id` | 预约 ID |
| `gate_type` | `进闸` 或 `出闸` |
| `truck_plate` | 识别车牌 |
| `container_no` | 识别箱号 |
| `check_result` | 校验结果 |
| `block_reason` | 拦截原因 |
| `created_at` | 记录时间 |

### 7. `exception_record`

用于记录异常闭环。

| 字段 | 含义 |
|---|---|
| `object_type` | `container`、`task`、`appointment` 等 |
| `object_id` | 对象 ID |
| `exception_type` | `箱号不符`、`残损`、`未放行`、`超窗` 等 |
| `status` | `待处理`、`处理中`、`已解除`、`已关闭` |
| `handler` | 处理人 |
| `resolution` | 处理结果 |

## 八、推荐状态机设计

### 1. 集装箱实物状态

```text
在船上
  -> 已卸船
  -> 转运中
  -> 在场
  -> 已预约锁定
  -> 已装车待出闸
  -> 已离港
```

异常分支：

```text
任一节点 -> 异常挂起 -> 异常解除 -> 回到原流程节点
```

### 2. 海关监管状态

```text
未申报 -> 查验中 -> 已放行
              -> 扣留 -> 已放行
```

闸口进场和预约确认必须要求监管状态为 `已放行`。

### 3. 预约状态

```text
待确认 -> 已确认 -> 已进闸 -> 已提箱 -> 已出闸
       -> 已取消
       -> 已过期
```

### 4. 任务状态

```text
pending -> in-progress -> completed
        -> suspended
        -> cancelled
        -> failed
```

建议当前 `task.next_status` 不再只做简单的 pending/in-progress/completed 递进，而是根据任务类型触发不同业务动作：

- `岸桥卸船` 完成：箱子变为 `已卸船`。
- `AGV转运` 开始：箱子变为 `转运中`。
- `场桥入堆` 完成：箱子变为 `在场` 并占用箱位。
- `堆场提箱` 完成：箱子变为 `已装车待出闸` 并释放箱位。
- `闸口出场` 完成：箱子变为 `已离港`。

## 九、对当前项目的优先改造路线

### 第一阶段：补齐进口闭环

1. 在 `container` 表增加 `customs_status`、`appointment_status`、`damage_status`。
2. 新增 `pickup_appointment` 表和简单预约接口。
3. 新增 `gate_transaction` 表和闸口校验接口。
4. 将 `container.status` 的最终流程补齐为 `在船上 -> 已卸船 -> 转运中 -> 在场 -> 已装车待出闸 -> 已离港`。

### 第二阶段：增强异常防呆

1. 新增 `exception_record`。
2. Manifest 导入时记录重复箱、缺失箱号、箱型非法、箱位非法。
3. 岸桥卸船时增加实际箱号确认。
4. 闸口进出时增加拦截原因。

### 第三阶段：增强资源调度和并发

1. 新增 `yard_slot`，用箱位锁避免重复占用。
2. 新增 `workflow_run`，持久化后台工作流状态。
3. 任务分配前锁定设备和箱位。
4. 对 Manifest 导入、预约确认、闸口出场使用数据库事务。

## 十、可用于答辩的总结

当前系统已经完成了进口箱前半段的核心数字化：Manifest 导入后能够生成船舶、集装箱和作业任务，并能模拟岸桥、AGV、场桥协同完成卸船入堆。这说明系统已经具备 TOS 的基础作业流转能力。

但要覆盖“从卸船到最终离港”的完整生命周期，还必须补齐监管放行、提箱预约、闸口校验、堆场提箱、异常闭环和并发资源锁定。优化后的系统不应只关心箱子在哪里，还要同时判断“箱子是否允许提、谁来提、什么时候提、从哪个闸口进出、异常是否处理完”。只有实物状态、监管状态和业务单证状态三者都闭环，进口箱流程才算真正完整。
